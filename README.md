# Smarter Flooding: Learning When to Relay in LoRa Mesh Networks
## Abstract
Meshtastic and similar LoRa mesh systems currently make retransmission decisions using simple heuristics such as hop limits, random backoff, duplicate suppression, and signal thresholds. These approaches are robust and decentralized, but largely context-blind. A node deep inside a dense urban cluster and a node acting as a sparse bridge at the edge of coverage may apply identical relay behavior despite occupying fundamentally different positions in the propagation field.
This paper proposes treating retransmission behavior as a learned probabilistic propagation problem rather than a routing problem. Instead of maintaining routes or disseminating topology state, the system periodically derives compact relay fingerprints from aggregated local telemetry and offline global propagation simulation. These fingerprints encode the long-timescale statistical relay utility of nodes within the surrounding mesh. During deployment, nodes apply these fingerprints locally when making retransmission decisions, without requiring route tables, topology synchronization, or coordination with neighboring nodes.
A globally aware simulator measures how much each node contributes to network-wide propagation coverage under varying relay conditions. The expensive optimization occurs offline, while deployed nodes perform only lightweight probabilistic relay behavior using compact stored parameters suitable for constrained hardware such as ESP32-based LoRa devices.
The result is a flooding system that preserves the robustness and graceful degradation properties of conventional broadcast propagation while using airtime more selectively.
## The Problem with Flooding, and Why Routing Doesn't Solve It
Anyone who has run Meshtastic in a busy area knows the tradeoff: Flooding burns through airtime. In dense deployments, a significant fraction of all transmissions are redundant: copies of something the recipient already received from three other nodes. In sparse deployments the problem inverts: suppression rules cut retransmissions that were actually needed.
The reason this is hard to fix is that the right retransmission decision depends on context that a node simply does not have. A node does not know whether other nodes already covered a downstream area. It does not know whether it sits on the only path to a sparse cluster two hops away, or whether it is one of six nodes all about to relay the same packet into an already-saturated zone. But this could be statisticall estimated from the data that the node does have.
The conventional answer to this problem is routing: establish paths, maintain topology, forward toward the destination. But LoRa environments make routing difficult. Links are asymmetric, intermittent, and load-dependent. Topology changes faster than it can be propagated. Airtime spent advertising routes is not spent carrying messages. Hikers enter and leave an area; one node going to sleep might change the entire map.
The proposal here is using machine learning to give each node a better local estimate of when retransmitting is actually useful.
## Framing the Question
Routing asks: where should this packet go next? Flooding asks: should I suppress this or not? The framing proposed here asks: how much does my retransmission improve network-wide propagation?
In routing, correctness means following the optimal path. In propagation, correctness means contributing to coverage. A retransmission that reaches three nodes nobody else would have reached is valuable. A retransmission that duplicates coverage already provided by two neighbors is not.
The system remains a broadcast system: packets go everywhere the relay chain reaches. The question the model answers is simply whether this nodes retransmission expands or improves that reach, or whether it mostly adds noise and congestion to an already-covered zone.
The practical consequence is a system that keeps the robustness properties of flooding, no routes to maintain, no topology to keep current, graceful degradation when nodes disappear, while spending airtime more deliberately.
## What Nodes Can Actually Observe
The constraint that shapes the entire architecture is that deployed nodes cannot have global knowledge. Disseminating topology information is expensive on LoRa. What a node can observe locally is:
- Signal strength and quality of packets it received
- How many neighbors it hears, and roughly how dense they are
- How many duplicate copies of this packet it has seen
- Its recent airtime usage and the apparent congestion level
- How many hops the packet has already traveled
- Whether it has recently heard retransmissions from nearby nodes for this packet
These signals could be used to approximate what relaying behavior would be valuable to the network. Nodes whose long-timescale telemetry consistently reflects dense duplicate-heavy conditions are likely to occupy well-covered regions of the propagation field. Nodes whose telemetry reflects sparse weak receptions and low neighborhood redundancy are likely to occupy propagation frontiers or bridge regions. A node surrounded by many active neighbors is probably in a congested urban canyon that does not need its retransmission as much as a node with few.
These local signals are predictive of propagation utility. Not perfectly, but well enough to make significantly better decisions than fixed rules. The model learns how long-timescale local propagation statistics relate to network-wide relay utility and compresses this into compact node-specific relay fingerprints.
## How the Training Works
The fundamental challenge is that propagation utility is not locally observable. A node cannot directly see how much coverage its retransmission would add. That depends on what all the other nodes do, the full graph geometry, and the resulting propagation across the network. This is why a training-time simulator is needed.
The oracle is a global simulator that runs during training only. It takes a network topology and simulates how a message propagates under different relay policies. A relay policy is simply: for each node, should it retransmit or not?
For any given nodes retransmission decision, the oracle can measure the counterfactual: how much does total network coverage change if this node retransmits versus suppresses? That difference — expressed as newly reached nodes, or improvement in reception probability across the network — becomes the training label. The oracle is answering the question the node cannot answer itself: given everything I know about the network, how much did this retransmission actually matter?
This is a standard simulation-to-deployment pattern. The expensive global computation happens once, offline. The deployed node uses a compact relay fingerprint periodically derived from aggregated local telemetry and offline propagation analysis. Training uses many generated topologies, source locations, and propagation scenarios to learn how long-timescale local propagation statistics relate to structural relay utility. The resulting fingerprints are topology-conditioned rather than topology-agnostic: they encode the persistent statistical role a node occupies within the propagation field of its surrounding mesh.
## What the Model Is Not Learning
If you train a model by asking it to imitate the next-hop decisions of a routing oracle, the model will learn to recognize structural roles in the graph. Node 8 tends to be a bridge. Nodes in the northeast quadrant tend to forward east. The model encodes a compressed version of the topology. This is fundamentally brittle: it depends on memorized relay corridors that do not generalize when the network changes.
The training signal here is different. The target is a scalar, not a forwarding decision. The inputs are local observations that do not include node identity or position. The system does not require explicit routing tables or stable next-hop paths, but it can still infer that certain nodes occupy persistently valuable relay positions within the propagation field through their long-timescale telemetry signatures. It learns relationships between aggregated local propagation statistics and long-timescale relay utility within the surrounding mesh. That knowledge can be periodically recomputed as the topology and propagation environment evolve.
## The Propagation Field
When a node retransmits, it extends the coverage field. When it suppresses, it lets the field propagate without it. The collective behavior of all nodes shapes the total field; how far coverage reaches, how dense it is in populated areas, how efficiently airtime is spent.
Each node accumulates long-timescale local propagation statistics from which the offline system estimates its persistent relay role within the field. A node that has seen many recent retransmissions nearby is probably inside a saturated zone. A node that heard a single weak copy is probably near the edge of the field. A node with many active neighbors probably does not need to contribute; a node in a sparse area probably does.
This is similar to how diffusion or epidemic processes work: nodes applying locally assigned probabilistic relay behaviors produce coherent global propagation behavior. The assigned relay fingerprints are tuned against simulated propagation outcomes rather than hand-designed heuristics or proxy metrics.
## Deployment Considerations
For this to work in Meshtastic hardware, the model needs to be small enough to run on constrained devices. The inference operation is straightforward: apply a lightweight probabilistic relay policy parameter derived offline from long-timescale local propagation statistics. This is well within the memory constraints of Meshtastic hardware.
The training pipeline runs entirely offline, on conventional hardware. Updated fingerprints would be small and could be distributed to devices via the network, or as firmware updates. Nodes do not coordinate during deployment; each applies its locally assigned relay fingerprint independently. The oracle does NOT fundamentally require real-world telemetry. Telemetry serves local observability of propagation conditions. Participating nodes could periodically contribute lightweight digests to the oracle via internet-connected nodes when airtime is detected to be abundant, for instance at night.
The system defaults gracefully to standard flooding behavior if the model is absent. It is strictly additive: a node running this system participates in the mesh normally and simply makes better-calibrated suppression decisions.
## Conclusion
Current Meshtastic flooding is robust because it requires nothing from nodes except a willingness to retransmit. Its weakness is that it does not adapt to context. Every retransmission decision is made with the same rules regardless of whether the node is a critical bridge to an isolated cluster or one of twelve nodes all about to relay the same packet into a congested urban zone.
The system proposed here keeps everything that makes flooding robust and adds a learned local estimate of when retransmitting is actually useful. The expensive global computation happens on hardware where it can afford to be thorough. The deployed model is cheap local inference. The communication is lightweight and not time-sensitive.
The key claim is that the local signals available to any Meshtastic node (e.g. duplicate counts, signal quality, neighbor density, airtime pressure) carry enough information to make significantly better-calibrated suppression decisions than current heuristics. Training a model against actual propagation outcomes, rather than route imitation, produces a system that learns to read these signals correctly.
The network still floods. It just floods intelligently.
